Technical Depth · Indexing

A Knowledge Graph that understands your game code

Indexing Pipeline
📥

Clone repo

Shallow clone (depth=1) for speed. One-time setup per repository.

🔍

GitNexus analyzes source

Parses all files → builds call graphs → detects communities → extracts execution processes.

🗄️

Graph loaded to Neo4j

Files as nodes, CALLS/IMPORTS as edges. Cached in memory for instant queries.

Why a knowledge graph?

Game code is deeply interconnected. A PR touching PlayerInventory might break CraftingSystem, SaveLoad, and UIManager — through 4 levels of calls. Flat file diffs miss this. The graph traverses the actual dependency chain to surface real risk.

Graph Query Tools

impact

Returns every system in the blast radius of a changed file.

detect_changes

Maps PR diff lines to the functions and modules they belong to.

context

Shows every caller and callee of a given function or file.

list_processes

Extracts end-to-end execution flows like player login or item purchase.

query

Semantic search over symbols and files by meaning, not just name.

Tool Budget

Each stage gets a fixed call budget (10–20 tool calls). If exhausted, a forced-submit agent captures partial results — no silent failures.

LangGraph interrupt() pauses execution at review points. QA can approve or refine — then the graph resumes from that exact state.

Smart LLM strategy

Heavy reasoning stages (mechanics, research, reproduction) use Claude Sonnet 4.6 for depth. Lightweight stages (triage, report synthesis) use Claude Haiku 4.5 — ~37× cheaper. Best quality where it counts, cost-efficient everywhere else.

MCP client lifecycle

A single MultiServerMCPClient is created at session start and reused across all 5 stages — avoiding repeated server spawns. Cleaned up on session close. Integrations like Grafana, Jira, and Notion are queried only in the research stage where they're relevant.

Why MCP matters for studios

Studios don't want another silo. They want their bug reports in Jira, their logs from Grafana, and their docs from Notion — all in one AI-assisted flow. MCP makes that possible without custom integrations per client.

Stage-aware tool filtering

Each agent stage only gets access to the tools it needs — preventing hallucinations and keeping tool context tight:

triage: jira_search, search_code, impact
mechanics: cypher, query, get_file_contents
research: grafana_*, kibana_*, jira_*, notion_*, sniffer_*
report: jira_create_issue, jira_update_issue
Technical Depth · Evaluation

Built to get better — not just run once

Human-in-the-loop as evaluation
1

Checkpoint reviews

QA explicitly approves or refines each stage — this is ground truth feedback on agent quality.

2

Feedback-driven reruns

When QA refines, the agent re-runs with explicit feedback injected — closing the loop within a session.

3

LangSmith evals

Offline evaluation suite in backend/evals/ — run against test cases to track prompt quality regression.

Confidence scores on outputs

Every bug report and component analysis includes a confidence level (high / medium / low) derived from how much evidence was gathered. Low-confidence results are explicitly flagged — false confidence is worse than no result.

high — multiple sources corroborate medium — partial evidence low — budget exhausted / sparse data
Budget exhaustion graceful fallback

If a stage hits its tool call limit without completing, a forced-submit agent captures all accumulated context and produces a partial result — with confidence marked low. No silent failures.

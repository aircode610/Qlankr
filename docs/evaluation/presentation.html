Context

What does the agent do?

📥
Input
A GitHub Pull Request URL — any repo, any size.
🤖
8-stage AI pipeline
Reads code, calls knowledge graph tools, generates unit + integration or E2E test plans.
📋
Output
Affected components, risk list, and ready-to-implement test specs.

The agent uses human-in-the-loop checkpoints — users review unit tests before choosing integration or E2E planning.

The Problem

Why "looks good" isn't enough

  • 🎲 Non-deterministic — the same PR produces different outputs each run
  • 📏 No ground truth — there's no single "correct" test plan to compare against
  • 🔗 Multi-step — a failure in stage 2 of 8 silently poisons all downstream output
  • 🔄 Regressions are silent — a small prompt change can quietly drop quality weeks later

Evaluation Design

15 evaluators across 3 layers

Layer 1 — Deterministic
Pipeline health
Did the agent complete? Did all stages run? Are all required fields present?

Free to run — pure code, no LLM cost.
Layer 2 — Deterministic
Behaviour checks
Did it use the right tools? Was it efficient? Did it use the knowledge graph when available?

Free to run — checks tool call lists.
Layer 3 — LLM Judge
Quality judgements
Did it catch the right risks? Are test specs useful? Are claims grounded in real code?

Scored by Claude — 0.0 to 1.0 with reasoning.
Layer 1 — Pipeline Health

Did the pipeline complete correctly?

binary
no_crash
"Did the agent finish without an error?"
Returns 1.0 if affected_components is non-empty in the output. Returns 0.0 if an error key is present. Returns 0.5 if it completed but found nothing.
ratio
pipeline_progression
"Did the agent reach all expected stages?"
Checks stages_seen against the expected list for the path taken.
Integration path expects: gather → unit_tests → integration_tests
Score = stages hit / stages expected.
avg
output_completeness
"Did every component have all 5 required fields filled?"
For each component: checks component name, files_changed, impact_summary, risks, confidence — scores filled/5, then averages across components.
ratio
component_count
"Did the agent find enough components?"
Score = min(1.0, found / expected_minimum). Each dataset example specifies a min_components threshold from human review.
Layer 2 — Behaviour

Did the agent behave correctly?

ratio
tool_coverage
"Did the agent call all the tools it should have?"
Score = |called ∩ expected| / |expected|. Each dataset example lists the tools a thorough analysis requires (e.g. get_file_contents, impact, submit_unit_tests).
penalty
tool_efficiency
"Did the agent waste tool calls by repeating itself?"
Counts any tool called more than 4 times as "redundant". Score = 1 − (redundant / total). Submit tools are excluded from this check.
binary
gitnexus_usage
"Did the agent use the knowledge graph when the repo was indexed?"
Only applies when expect_gitnexus=True in the dataset. Checks that at least 2 of impact, context, query, cypher were called. Returns N/A (1.0) for GitHub-only repos.
ratio
confidence_calibration
"Is the agent's confidence level realistic?"
For non-indexed repos: expects all components to have confidence="low" (no graph data). For indexed repos: expects at least half to be medium or high.
Layer 3 — LLM Judges

Did the agent produce quality output?

judge
risk_quality
"Did the agent catch the risks a human reviewer identified?"
Claude is given the agent's risks and the golden expected risks. It scores recall (were the expected concerns found?), precision (were extra risks relevant?), and penalises missed critical ones. Exact wording doesn't matter — semantic match counts.
judge
component_matching
"Did the agent name components the same way a human would?"
"ReAct Agent Core" should match "ReAct Agent". Exact string comparison would fail — Claude judges semantic equivalence between the agent's names and the golden names.
judge
groundedness
"Are the agent's claims backed by real code changes?"
Claude is given the PR URL and the full analysis. It asks: "could a human reading the same PR arrive at these claims?" Penalises hallucinated file names, invented risks, or fabricated symbols.
judge
unit_test_quality & integration_test_quality
"Are the generated test specs actually useful to a developer?"
Claude checks: Are targets specific real symbols? Do test cases cover edge cases? Are mocks appropriate? Are integration points real cross-module boundaries? Returns 0.0–1.0 with reasoning.

Interpreting Results

What the scores tell us

✅
Pipeline is solid
no_crash, pipeline_progression, output_completeness, component_count all at 1.00. The agent reliably completes the full pipeline and finds at least the minimum expected components on every PR.
✅
Tool usage is healthy
tool_efficiency 0.89 means the agent rarely repeats itself. tool_coverage 0.86 means it usually calls the right tools — only missing some on simpler PRs where fewer tools are needed.
⚠️
Risk detection needs improvement
risk_quality 0.66 and groundedness 0.72 — the agent finds roughly 2/3 of the risks a human reviewer would flag. It misses subtle edge cases and sometimes introduces weakly-grounded claims.
🔴
Integration tests are the weakest point
integration_test_structure 0.29 and quality 0.18 — the agent often exhausts its tool budget on research without calling submit. Confidence calibration 0.18 shows it's also rating non-indexed repos too highly.
